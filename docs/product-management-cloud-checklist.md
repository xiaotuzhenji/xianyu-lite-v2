# 商品管理云端验证清单

> 只在云服务器执行，不在本地启动 Docker、前端或后端。

## 部署前检查

- 仓库内不得提交服务器 IP、账号密码、Cookie、Token。
- 云服务器使用 `.env` 或服务器环境变量保存敏感配置。
- 执行前确认 Docker 与 Docker Compose 在服务器可用。

## 基础启动检查

```bash
sh scripts/cloud-product-check.sh
```

- 脚本会先构建后端镜像，再在容器内运行后端规则测试，避免使用旧镜像。
- 脚本应在云服务器 Linux shell 中执行；仓库已约束 `*.sh` 使用 LF 换行。
- 商品管理相关后端测试应全部通过。
- `frontend`、`backend`、`mysql`、`scheduler`、`websocket` 均应处于运行状态。
- 后端健康检查应返回成功。
- 如健康检查失败，先执行 `docker compose logs backend --tail=100` 查看原因。
- Web 页面可正常登录，无接口返回 `Internal Server Error` 或非 JSON 响应。
- 如果后端或反向代理异常返回 HTML/纯文本，前端应展示可读错误，不应出现 `Unexpected token` 这类 JSON 解析报错。

## 商品同步检查

- 选择单个账号同步商品，应返回同步数量。
- 商品页账号下拉只能显示当前用户自己的账号。
- 多账号用户只能查看、同步、编辑、发布、下架自己账号下的商品。
- 同步时如果发现商品 ID 已属于其他账号，应跳过并提示错误，不应改写到当前账号。
- 同步已有在售商品时，不应额外生成草稿商品。
- 同步后商品状态应显示为“已上架”。
- 多图商品同步后，应保留多张图片，而不是只剩一张。
- 平台已下架的商品，再同步后应标记为“已下架”。

## 草稿发布检查

- 没有可用账号时，点击“新建草稿”也应打开弹窗，并在保存时提示先选择或添加账号，不能无反应。
- 新建草稿标题最多 30 个字。
- 商品价格不能为负数，前端应提示，后端也应拒绝保存。
- 连续快速新建多个草稿，不应出现商品 ID 冲突。
- 商品 ID 不应作为普通可编辑字段，避免手动改错目标商品。
- 新建商品只能成为草稿，不能通过前端或接口参数直接伪造成“已上架”。
- 草稿可上传多张图片，并在保存后继续显示。
- 商品图片最多 8 张，超过数量时前端应提示，后端也应拒绝保存。
- 删除一个草稿时，如果图片仍被其他商品引用，不应导致其他商品图片失效。
- 点击“发布到闲鱼”后应进入“发布中”状态。
- 发布成功后应显示“已上架”，并使用平台返回的新商品 ID。
- 如果发布结果未拿到平台商品 ID，应显示失败原因，不应把草稿或旧商品标成“已上架”。
- 发布失败时应显示明确失败原因，不应一直卡在“发布中”。

## 已上架商品重发检查

- 已上架商品编辑后可以重新发布。
- 编辑标题、价格、描述、图片时，不应直接改变商品“已上架/草稿/下架/发布中”状态。
- 重新发布轮询时，不应把旧商品原本“已上架”的状态立即误判为发布成功。
- 重新发布成功后，新商品应显示为“已上架”。
- 旧商品应被下架；如果下架失败，应在页面展示错误提示。
- 已上架商品不应直接删除，必须先下架。

## 发货配置检查

- 每个具体商品可以单独配置发货内容。
- 发货配置只能读取、保存、删除当前用户自己账号下的商品配置。
- 保存后重新打开同一商品，应能读回该商品自己的配置。
- 不同商品之间的发货配置不应互相覆盖。

## 通过标准

- 页面无乱码、按钮状态清楚、失败原因可见。
- 商品同步、草稿发布、已上架重发、多图、下架、删除、发货配置均按预期工作。
- 后台日志无连续异常，页面无非 JSON 报错。
